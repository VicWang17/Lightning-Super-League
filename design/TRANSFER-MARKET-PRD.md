# 转会市场系统 PRD

> 版本：v0.2  
> 日期：2026-06-04  
> 范围：队间转会报价、球员挂牌、报价公示、AI交易决策、解约违约金、转会历史。  
> 状态：已按当前代码结构做可操作性修正，待确认后进入技术设计与实现。

## 1. 背景与目标

当前项目已经具备以下基础：

- 财务系统：`TeamFinance`、`TeamSeasonFinance`、`FinanceTransaction`，转会收支可进入 `TRANSFER` 流水。
- 合同系统：`ContractService` 支持签约、续约、解约，球员有工资、合同到期赛季、阵容角色、状态分。
- 自由市场：已有 `FreeAgentListing` 和自由市场签约入口。
- AI运营：AI球队会做续约、青训签约、自由市场补人，但还缺少主动买卖和处理报价逻辑。
- 前端页面：已有转会市场、自由市场、我的挂牌、历史页壳子。
- 当前代码约束：`ContractService.ROSTER_MAX = 18`，`ROSTER_MIN = 8`；`SquadRole` 字段存在，但目前没有稳定的“关键球员/一线球员”产品判定，转会 v1 不能依赖它做核心规则。

本次目标是补齐“有队球员”的市场流转：

```
玩家/AI查看球员
  -> 对其他球队球员报价
  -> 对方接受/拒绝/反报价/超时自动处理
  -> 成交后资金结算、球员转队、合同继承或重签
  -> 交易公开、公示、历史可查

本队球员
  -> 挂牌
  -> 多队报价
  -> 玩家/AI决策或超时自动按规则处理
  -> 成交/撤牌/过期

本队球员
  -> 直接解约
  -> 支付较高违约金
  -> 球员进入自由市场
```

## 2. 设计原则

1. **市场透明**
   - 玩家发出的队间报价全服可见。
   - 成交价、系统估值、买卖双方、是否挂牌、是否反报价都进入转会历史。

2. **少量但有重量的操作**
   - 玩家每天只有 2 次主动报价机会。
   - 解约成本必须明显，避免玩家把解约当作免费清仓工具。

3. **保护阵容下限和财政秩序**
   - 成交、解约、AI自动卖人都不能让球队低于一线队人数下限。
   - 购买方必须通过余额、转会预算、工资压力检查。

4. **AI要像俱乐部，不像随机数**
   - AI首先保证阵容完整和财政健康，再根据位置短缺、冗余、球员价值、预算做买卖。
   - AI可以较快响应，但仍允许少量延迟制造真实市场节奏。

5. **先做队间报价，不做完整拍卖**
   - 早期 PRD 有公开拍卖体系。本版优先实现更贴近你新需求的“报价/反报价/挂牌等待期”。
   - 未来可以把挂牌页升级为拍卖，但 v1 不引入代理出价、倒计时保护、资金冻结竞拍等复杂机制。

## 3. 术语

| 术语 | 说明 |
| --- | --- |
| 市场估价 | 系统基于 OVR、年龄、潜力、状态、联赛级别等计算的参考价。 |
| 挂牌 | 拥有球队主动标记球员可出售，并设置挂牌价。 |
| 主动报价 | 买方向非本队球员发送的购买报价。 |
| 报价链 | 围绕同一名球员、同一买卖双方的一轮有限讨价还价。最多包含初始报价、反报价、最终报价三步。 |
| 初始报价 | 买方主动发送的第一份报价，占用每日主动报价次数。 |
| 反报价 | 卖方对初始报价发起的唯一一次抬价回应。 |
| 最终报价 | 买方收到反报价后最后一次出价，卖方只能接受或拒绝。 |
| 等待期 | 报价/挂牌后 3 天的决策窗口。 |
| 公开报价 | 所有球队/玩家可见的报价记录。 |
| 成交 | 报价被接受或超时自动接受后，执行资金和球员归属变更。 |

## 4. 功能范围

### 4.1 v1 必做

- 玩家每天最多发送 2 次主动报价。
- 玩家可对其他球队非退役、有合同球员发送报价。
- 接收球队必须在 3 天内接受、拒绝或反报价。
- 同一报价链最多讨价还价一次：买方初始报价 -> 卖方反报价 -> 买方最终报价 -> 卖方接受/拒绝。
- 挂牌球员三天不回复自动同意最高有效报价。
- 非挂牌球员三天不回复自动拒绝。
- 每有新报价进入某个挂牌，等待期自动续期 3 天。
- 玩家可挂牌本队球员，挂牌价不得低于市场估价的 80%。
- 玩家可直接解约本队球员，但支付违约金。
- AI可处理收到的报价、发起报价、挂牌冗余球员、接受/拒绝/反报价。
- 成交写入双方财务流水和转会历史。
- 报价、反报价、成交、拒绝、超时处理进入邮件/通知。

### 4.2 v1 不做

- 租借。
- 分期付款、二次分成、附加条款。
- 经纪人、球员个人谈判拒绝转会。
- 完整拍卖倒计时和代理竞价。
- 真实转会窗限制。v1 默认全赛季开放，后续可接入赛季时间线。
- 恶意小号交易检测的完整处罚系统。v1 仅记录异常标记。

## 5. 市场估价

### 5.1 估价公式

建议沿用并收敛早期 PRD 的估值口径，但适配当前项目经济尺度。当前 `Player.market_value` 过于简化，只适合作临时展示，不适合作交易底价。

```
市场估价 =
  OVR基础价
  * 年龄系数
  * 潜力系数
  * 状态系数
  * 联赛级别系数
  * 位置稀缺系数
```

### 5.2 建议系数

OVR基础价：

| OVR | 基础价 |
| --- | --- |
| 30 | 20万 |
| 40 | 60万 |
| 50 | 150万 |
| 60 | 350万 |
| 70 | 800万 |
| 80 | 1800万 |
| 90 | 4000万 |

未命中用线性插值。

年龄系数：

| 年龄 | 系数 |
| --- | --- |
| <=20 | 1.20 |
| 21-23 | 1.10 |
| 24-26 | 1.00 |
| 27-28 | 0.85 |
| 29-30 | 0.65 |
| 31-32 | 0.45 |
| >=33 | 0.30 |

潜力系数：

| 潜力 | 系数 |
| --- | --- |
| S | 1.50 |
| A | 1.25 |
| B | 1.10 |
| C | 1.00 |
| D | 0.85 |

状态系数：

| 状态 | 系数 |
| --- | --- |
| HOT | 1.15 |
| GOOD | 1.05 |
| NEUTRAL | 1.00 |
| LOW | 0.90 |
| INJURED/SUSPENDED | 0.75 |

联赛级别系数：

| 级别 | 系数 |
| --- | --- |
| 1 | 1.25 |
| 2 | 1.10 |
| 3 | 1.00 |
| 4 | 0.85 |

位置稀缺系数：

- 默认 1.00。
- 当前活跃挂牌/自由市场中某位置供应不足时，上浮到 1.05-1.15。
- 供应明显过剩时，下调到 0.90-0.95。
- v1 可以先固定 1.00，后续再接入市场供需。

## 6. 主动报价规则

### 6.1 报价资格

买方必须满足：

- 当天主动报价次数 < 2。
- 球队不处于 `restricted/crisis` 等禁止转会状态。
- 报价金额不高于当前余额，且不高于可用转会预算加允许透支额度。
- 成交后名单人数不超过上限。
- 若需要新合同或工资继承，工资压力必须在允许范围内。

目标球员必须满足：

- 非本队球员。
- `status` 不是 `RETIRED`。
- 有所属球队。
- 不处于其他已接受待结算交易中。
- 不违反新秀/刚入队冷却期。建议 v1 冷却期为 3 天。

### 6.2 报价金额限制

- 对挂牌球员：首次报价必须 >= 挂牌价。
- 对非挂牌球员：首次报价建议不得低于市场估价的 80%。
- 低于估价 90% 的非挂牌报价，AI大概率直接拒绝。
- 高于估价 250% 的报价标记为异常交易，但 v1 不阻止，除非余额/预算不满足。

### 6.3 报价公开

玩家发送的报价所有人可见，至少展示：

- 球员、位置、OVR、年龄。
- 买方球队、卖方球队。
- 报价金额。
- 市场估价。
- 是否挂牌。
- 当前状态：待回复、被反报价、最终报价待回复、已接受、已拒绝、已过期、已成交。
- 截止时间。

AI发送的报价建议也公开，保持市场透明和统一。

### 6.4 接收方决策

收到初始报价时，卖方可执行：

- 接受：立即进入成交结算。
- 拒绝：报价结束。
- 反报价：只能对该报价链使用一次，金额必须高于原报价，建议不超过原报价的 150%。
- 不处理：等待超时自动规则。

收到反报价时，原买方可执行：

- 接受反报价：按卖方反报价金额成交。
- 拒绝反报价：报价链结束。
- 提交最终报价：最后一次加价，之后卖方只能接受或拒绝。

收到最终报价时，卖方可执行：

- 接受：按最终报价成交。
- 拒绝：报价链结束。
- 不允许再次反报价。
- 不处理：等待超时自动规则。

### 6.5 反报价规则

每个报价链最多三步：

```
Step 1: 买方 initial_offer
  -> 卖方 accept / reject / counter

Step 2: 卖方 counter_offer
  -> 买方 accept_counter / reject / final_offer

Step 3: 买方 final_offer
  -> 卖方 accept / reject
```

规则：

- 卖方只有一次反报价机会。
- 买方只有一次最终报价机会。
- 最终报价必须高于初始报价，且不得高于卖方反报价；如果买方愿意支付反报价，应直接使用“接受反报价”。
- 初始报价、反报价、最终报价都公开展示。
- 初始报价占用买方每日主动报价次数。
- 最终报价也占用买方每日主动报价次数；若当天次数不足，买方仍可接受反报价或拒绝，但不能提交最终报价。
- 每一步响应窗口都是 3 天。
- 任一步超时：
  - 球员当前处于挂牌状态：自动接受该报价链内“当前对卖方有效的最高买方报价”。如果只有卖方反报价且买方未接受，也没有最终报价，则报价链过期。
  - 球员当前不是挂牌状态：自动拒绝。
- 若球员挂牌期间出现其他球队的新报价，挂牌等待期按新报价续期。原报价链仍保留，但自动成交时只比较所有可成交的买方报价，不把卖方反报价当作买方报价。

## 7. 挂牌规则

### 7.1 挂牌资格

球队可以挂牌本队球员，但必须满足：

- 球员属于本队，非退役。
- 球队挂牌后仍可满足最低可用名单要求。
- 球员不在已接受待结算交易中。
- 距离上次转入/续约/挂牌撤回超过冷却期。建议 v1 为 3 天。

### 7.2 挂牌价

- 系统给出市场估价作为参考。
- 挂牌价不得低于市场估价的 80%。
- 挂牌价可以高于估价，不设硬上限，但高于 200% 时前端给出“可能无人报价”的提示。

### 7.3 挂牌等待期

- 首次挂牌后创建 3 天等待期。
- 每收到一份新的有效报价，等待期自动续期到该报价创建时间 + 3 天。
- 玩家/AI可随时接受某一份有效报价并提前结束挂牌。
- 如果等待期结束仍未处理：
  - 有报价：自动接受最高有效报价。
  - 无报价：挂牌保持 active 但不再有自动成交倒计时，直到收到新报价或卖方撤牌。

### 7.4 多报价处理

同一挂牌可收到多队报价：

- 列表按金额降序展示。
- 相同金额时，先报价者优先。
- 最高报价变更时向卖方发通知。
- 自动成交时按最高金额成交。
- 未成交报价统一标记为 `outbid_closed`。

## 8. 非挂牌球员报价

非挂牌报价更像“询价+挖人”：

- 等待期 3 天。
- 接收方不处理时自动拒绝。
- 接收方可以按 6.5 规则反报价一次。
- AI对非挂牌高保护球员有明显保护，除非报价显著溢价或财政压力很大。高保护球员不是固定字段，按队内 OVR 排名、位置稀缺、合同工资压力实时计算。
- 玩家收到非挂牌报价时，前端应强调“该球员未挂牌，超时将自动拒绝”，避免误解。

## 9. 成交结算

### 9.1 成交前校验

接受或自动成交前必须再次校验：

- 买方余额仍足够。
- 买方 roster 未满。
- 卖方出售后不低于 roster 下限。
- 球员仍属于卖方。
- 球员没有退役或已被其他交易成交。
- 报价链仍为 `open`，当前报价仍为 `pending_response`。

若校验失败：

- 报价标记为 `settlement_failed`。
- 双方收到通知。
- 不发生资金和球员变更。

### 9.2 资金流

成交价全额从买方扣除，卖方收入可按规则扣除交易税。

建议 v1：

- 买方支出：成交价。
- 卖方收入：成交价 * 95%。
- 系统交易税：5%，仅作为财务流水描述，不进入任何球队。

财务流水：

- 买方：`TRANSFER / EXPENSE / amount=成交价`
- 卖方：`TRANSFER / INCOME / amount=成交价*0.95`

### 9.3 合同处理

建议 v1 默认采用“合同随球员转移”：

- 转会后球员保留原合同工资、合同到期赛季、阵容角色可重置为 `rotation` 或由买方确认。
- 买方必须承担原工资。
- 若买方希望续约/改工资，使用现有合同系统另行操作。

原因：

- 当前合同系统已经有工资满意度和合同历史。
- v1 若加入球员个人谈判，会显著放大实现范围。
- 合同随转让可以让交易市场先跑起来。

后续可升级为“成交后必须谈个人合同”，提高拟真度。

### 9.4 球员状态影响

转会成交后：

- 球员 `team_id` 更新为买方。
- 写入合同历史或转会历史。
- `PlayerStateService.recalculate_player_state(source_event="transfer_completed")`。
- 新队第一天可给轻微适应惩罚：`morale_score -1`，持续 3 天；v1 可先不做。

## 10. 解约规则

### 10.1 解约资格

玩家和AI可以解约本队球员，但必须满足：

- 球员属于本队。
- 球队解约后仍不低于 roster 下限。
- 球员不是已接受待结算交易的对象。
- 球队余额足以支付违约金。若不足，不允许解约。

### 10.2 未发工资计算

当前工资语义更接近“赛季工资”。建议：

```
未发工资 =
  当前赛季工资 * (当前赛季剩余天数 / 赛季总天数)
  + 后续完整合同赛季数 * 当前赛季工资
```

示例：

- 当前第 10/42 天，合同本赛季末到期，赛季工资 100万：
  - 未发工资 = 100万 * 32/42 = 76.19万。
- 当前第 10/42 天，合同还有本赛季 + 2 个完整赛季，赛季工资 100万：
  - 未发工资 = 76.19万 + 200万 = 276.19万。

### 10.3 违约金比例建议

为了让玩家“不忍心随意解约”，建议 v1 使用：

```
普通合同基础违约金 = 未发工资 * 75%
新人合同基础违约金 = 未发工资 * 50%
普通合同最低违约金 = 市场估价 * 10%
新人合同最低违约金 = 市场估价 * 5%
最终违约金 = max(基础违约金, 最低违约金)
```

修正项：

- 不使用 `KEY_PLAYER` / `FIRST_TEAM` 等角色提高违约金，因为当前系统没有可靠的关键/一线球员判定。
- 合同只剩当前赛季最后 7 天时，普通合同基础比例从 75% 降到 50%，避免赛季末清理成本过于粗暴。
- `NOT_NEEDED` 角色不降低违约金，因为玩家可以先把球员改成不需要再解约，容易被滥用。

这个方案的手感：

- 高薪长约球员解约会非常疼，符合现实和经营约束。
- 低薪低价值球员仍有成本，但不会卡死玩家。
- 老将/低OVR球员市场估价低，主要由剩余工资决定。
- 新秀试错成本较低，但不再是完全免费。

实现注意：

- 当前 `ContractService.release_player` 会直接把球员变为自由身，且不扣钱、不创建自由市场 listing。v1 需要新增 `TransferService.release_player_with_penalty`，不要直接复用旧接口作为玩家入口。
- 旧接口可以保留为内部/开发接口，或改为调用新方法并要求先预览违约金。

### 10.4 解约后流向

解约成功后：

- 支付违约金，写 `TRANSFER / EXPENSE` 或 `PENALTY / EXPENSE`。建议用 `TRANSFER`，便于转会支出统计。
- 旧合同标记 `terminated`。
- 球员 `team_id = null`、`contract_type = FREE`、工资归零。
- 创建 `FreeAgentListing(origin=RELEASED)`。
- 自由市场签字费建议为市场估价的 55%，沿用现有自由市场口径。

## 11. AI 行为策略

### 11.1 AI球队画像

每支AI球队可按隐藏策略分型：

| 类型 | 倾向 |
| --- | --- |
| balanced | 买卖均衡，重视阵容完整。 |
| contender | 更愿意溢价买关键位置，不轻易卖高保护球员。 |
| developer | 更愿意买年轻潜力、卖老将。 |
| frugal | 报价保守，财政健康优先。 |
| seller | 财政压力下更愿意挂牌/接受报价。 |

v1 可以先不落库，按球队财务健康、联赛排名、阵容结构动态计算倾向。

### 11.2 AI评估球员价值

AI内部价值：

```
AI价值 =
  市场估价
  * 位置需求系数
  * 队内保护系数
  * 年龄规划系数
  * 财政压力系数
  * 性格/状态轻微系数
```

位置需求系数：

- 本队该位置短缺：1.15-1.30。
- 人数充足：1.00。
- 明显冗余：0.80-0.90。

队内保护系数：

- 队内 OVR 排名前 3：1.30。
- 本位置人数 <= 目标人数：1.15。
- 本位置唯一门将或唯一可用球员：禁止主动出售，除非 roster 兜底逻辑可先补人。
- 队内 OVR 排名后 30% 且本位置冗余：0.85。
- 工资高于队内中位工资 1.5 倍但 OVR 排名后 50%：0.80。

说明：

- v1 不依赖 `SquadRole` 判断核心/一线球员。
- 如果后续加入真实出场时间、队长、承诺角色，再把这些因素加入保护系数。

财政压力系数：

- 健康 A/B：卖方不急，1.00。
- 健康 C：接受阈值降低 10%。
- 健康 D 或 overspend：接受阈值降低 20%-35%。

### 11.3 AI处理收到的报价

对挂牌球员：

- 报价 >= 挂牌价 且 >= AI内部价值 * 0.95：接受概率高。
- 报价略低于期望：反报价到 `max(挂牌价, AI内部价值*1.05)`。
- 报价过低：拒绝。
- 若多报价，优先最高报价。
- AI响应时间：建议 5-30 分钟内处理；开发环境可配置为立即。

对非挂牌球员：

- 高保护球员：低于 AI内部价值 * 1.35 基本拒绝。
- 轮换/替补：高于 AI内部价值 * 1.05 可考虑接受。
- 财政压力大：阈值下调。
- 位置冗余：阈值下调。
- 若报价有诚意但不足：反报价一次。

### 11.4 AI主动报价

AI每天可进行少量主动报价，避免市场过热：

- 每支AI每天最多 1 次主动报价。
- 只在以下情况触发：
  - roster 低于目标人数。
  - 某位置低于目标配置。
  - 球队财务健康良好且有明显升级机会。
  - 看到挂牌价低于 AI内部价值。
- 优先买挂牌球员，其次非挂牌低保护球员，避免频繁骚扰高保护球员。

### 11.5 AI挂牌球员

AI会挂牌：

- 位置明显冗余球员。
- 工资高但上场价值低的球员。
- 合同即将到期且不计划续约的球员。
- 财政压力下的低保护球员。

AI挂牌价：

- 普通情况：市场估价 * 1.00-1.20。
- 急售：市场估价 * 0.80-0.95。
- 年轻潜力球员：市场估价 * 1.20-1.50。

### 11.6 AI解约

AI不应频繁解约。只在：

- roster 超过软上限。
- 球员长期低价值、高工资、无人报价。
- 财务健康 D/crisis。
- 违约金低于保留工资压力的长期风险。

AI解约前应优先尝试挂牌，至少经过一个挂牌周期。

## 16. 闭环与可操作性审查

### 16.1 报价链闭环

报价链有明确终态，不允许无限讨价还价：

| 当前状态 | 可选动作 | 下一状态 |
| --- | --- | --- |
| 初始报价待卖方响应 | 接受 | 成交结算 |
| 初始报价待卖方响应 | 拒绝 | `rejected` |
| 初始报价待卖方响应 | 反报价 | 反报价待买方响应 |
| 反报价待买方响应 | 接受反报价 | 成交结算 |
| 反报价待买方响应 | 拒绝 | `rejected` |
| 反报价待买方响应 | 最终报价 | 最终报价待卖方响应 |
| 最终报价待卖方响应 | 接受 | 成交结算 |
| 最终报价待卖方响应 | 拒绝 | `rejected` |

所有待响应状态都有 `expires_at`。超时处理：

- 非挂牌球员：自动拒绝。
- 挂牌球员：只自动接受真实买方报价中的最高有效报价。卖方反报价不是买方报价，不能被自动成交。
- 如果挂牌球员只有“初始报价 -> 卖方反报价”，买方没有接受也没有最终报价，超时不能按卖方反报价成交，只能让该报价链过期。

### 16.2 代码实际落点

- `backend/app/models/transfer.py`：`TransferListing`、`TransferNegotiation`、`TransferOffer`、`TransferDailyQuota`、`TransferRecord`。
- `backend/app/services/transfer_service.py`：转会主流程、估价、报价链、成交、解约违约金。
- `backend/app/services/ai_transfer_service.py`：AI报价响应、市场扫描、挂牌决策。
- `backend/app/routers/transfers.py`：转会API。

- `backend/app/core/events.py`：新增转会事件枚举。
- `backend/app/services/season_service.py`：新增转会事件分发。
- `backend/app/main.py` / router 注册处：挂载 `transfers` 路由。

当前项目可复用：

- `MailCategory.TRANSFER` 写转会通知。

- `ContractService.ROSTER_MAX=18`、`ROSTER_MIN=8` 作为 v1 默认名单上下限。

### 16.3 合同随转移实现口径

v1 采用合同随球员转移：

- 更新 `players.team_id = buyer_team_id`。
- 保留 `players.wage`、`contract_end_season`、`contract_type`、`release_clause`。
- 同步更新该球员 active `player_contracts.team_id = buyer_team_id`，避免合同仍归属原球队。
- 转会前后的球队关系由 `transfer_records.from_team_id/to_team_id` 保留。
- 买方如果想改工资或年限，走现有续约接口。

### 16.4 财务与名单闭环

成交和解约都必须在最终提交前做二次校验：

- 买方余额 >= 成交价。
- 买方人数 < `ROSTER_MAX`。
- 卖方出售/解约后人数 >= `ROSTER_MIN`。
- 球员当前仍属于卖方。
- active 合同仍存在或球员合同字段可被同步。
- 对应报价/挂牌仍是 active/open 状态。

`FinanceService.can_sign_player` 当前主要检查余额，不严格阻止工资帽超限。转会服务需要单独检查成交价、余额、转会预算和 overspend 状态；工资帽规则按现有经济系统口径先做“预警/惩罚”，不在 v1 硬阻止，除非球队处于 `crisis`。

### 16.5 时间与额度闭环

- “每天 2 次报价”按 `Season.current_day` 统计，不按现实自然日。
- 初始报价和最终报价都消耗额度，因为二者都是买方向其他球队发出的报价。
- 接受反报价、拒绝、接受收到的报价不消耗额度。
- 3 天等待期使用 `EventQueue.scheduled_at` 动态事件实现；创建报价时按当前虚拟时间 `clock.now() + timedelta(days=3)` 入队。
- 快进/模拟时，事件队列会自然处理到期报价，不需要额外定时器。

### 16.6 当前PRD已闭合的终点

- 成交：资金流 + 球员归属 + active 合同归属 + 转会记录 + 通知。
- 拒绝：报价链终态 + 通知。
- 超时：挂牌按最高有效买方报价成交，非挂牌拒绝。
- 解约：违约金支出 + 合同终止 + 球员自由身 + 自由市场 listing + 通知。
- AI：收到报价可响应；每日扫描可买入、挂牌、少量主动报价；解约只作为最后手段。

## 17. 风险与边界

1. **名单上限不一致**
   - 早期PRD写 15 人，当前 `ContractService.ROSTER_MAX=18`，AI目标为16/17。
   - v1 开发默认按代码现状 18/8 执行；如产品决定改回 15，需要先同步修改合同、AI、roster lifecycle。

2. **工资帽规则目前偏宽**
   - `can_sign_player` 主要检查余额，不严格阻止工资帽超限。
   - 转会成交如果继承工资，可能加剧工资帽压力。需要决定是“允许但财务惩罚”还是“硬阻止”。

3. **合同随队转移的拟真度**
   - v1 为了闭环简单建议合同随转移。
   - 如果你想更拟真，需要成交后加入个人合同谈判，会显著增加范围。

4. **公开报价可能让市场信息过载**
   - 需要分页、筛选和默认只展示最近/高价值报价。

5. **AI交易过强或过弱**
   - 需要用闭环测试观察：AI是否卖空阵容、是否囤积球员、是否过度溢价。

## 18. 验收标准

- 玩家同一赛季日最多只能发送 2 次主动报价。
- 挂牌价低于估价 80% 时无法提交。
- 挂牌球员收到多份报价后，等待期按最新报价续期 3 天。
- 挂牌等待期结束时自动接受最高有效报价。
- 非挂牌报价等待期结束时自动拒绝。
- 每个报价链最多只有 1 次卖方反报价和 1 次买方最终报价。
- 买方提交最终报价后，卖方只能接受或拒绝，不能再次反报价。
- 成交后买方扣款、卖方入账、球员转队、财务流水正确。
- 成交后 active `player_contracts.team_id` 同步为买方球队。
- 解约会预览并扣除违约金，球员进入自由市场。
- AI能处理收到的报价，并能基于阵容和财政做基础买卖。
- 所有报价和成交记录可在公开报价/历史中查看。
